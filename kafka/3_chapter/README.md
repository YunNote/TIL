### 🚀 카프카의 기본 개념

카프카 브로커는 클라이언트와 데이터를 주고받기 위해 사용되는 주체이자, 분산 저장하여 장애가 발생하더라도 안전하게 사용할 수 있도록 도와주는 애플리케이션이다.

하나의 서버에는 하나의 카프카 브로커 프로세스가 실행되어 한대의 서버만으로도 실행이 되지만 안전하게 보관하고 처리하기 위해 기본적으로 3대 이상의 브로커 서버를 1개의 클러스터로 묶어서 운영한다.

---

### 🚀 카프카의 데이터 저장과 전송.

프로듀서로 부터 데이터를 전달받으면 카프카 브로커에서는 프로듀서가 요청판 토픽의 파티션에 데이터를 저장하고 컨슈머가 데이터를 요청하면 저장된 데이터를 저장한다.

카프카에서 데이터의 저장은 파일 시스템에 저장된다.

실제로 카프카 실행시 읽어들이는 설정파일 config/server.properties에서 `log.dir`에 설정된 경로에 파일시스템이 저장된다.

> 확인하는 방법은 topic에 데이터를 넣게 되면 해당 topic 명으로 생성되어 있음을 확인 가능


해당 폴더에 보면 `index`, `log`, `timeindex`등의 값을 볼 수 있는데.

- log: 메시지와 메타데이터를 저장한다.
- index: 메시지의 오프셋을 인덱싱한 정보를 담은 파일이다.
- timeindex: 메시지에 포함된 timestamp값을 기준으로 인덱싱한 정보가 담겨있다.
  > 0.10.0.0 버전 이후의 카프카에서는 메시지에 timestamp값이 포함된다. timestamp 값은
  > 브로커가 적재한 데이터를 삭제하거나 압축하는데 사용한다.

카프카는 기본적으로 파일 시스템에 저장하기 때문에 파일 입출력으로 인한 속도 이슈가 발생하지 않을까? 라는 의문을 품을 수 있따. 하지만 카프카는 페이지캐시를 사용하여 디스크 입출력 속도를 높여서 이 문제를
해결하였다


> #### 페이지 캐시란
>
> OS에서 파일 입출력의 선능 향상을 위해 만들어 놓은 메모리 영역을 의미한다.
> 한번 읽은 데이터에 대해서 메모리의 페이지 캐시 영역에 저장후 동일한 파일의 접근시 메모리에서 읽어내는 방식이다

### 🚀 데이터 복제, 싱크

데이터 복제는 카프카를 장애허용시스템으로 동작하도록 하는 원동력이다.

복제의 이유는 클러스터로 묶인 브로커 중 일부에 장애가 발생하더라도 데이터를 유실하지 않고 안전하게 사용하기 위함이다.

> 카프카의 데이터 복제는 파티션 단위로 이루어진다. 토픽을 생성할 때 파티션의 복제 개수도 같이 설정되는데 직접 옵션을 선택하지 않는다면 브로커에 설정된 값을 따라간다.
>
> 복제 개수의 최솟값은 1이고 최댓값은 브로커 개수만큼 설정하여 사용할 수 있다.


![img.png](img.png)

위의 이미지는 토픽의 파티션의 복제 개수가 3인 경우에 대한 이미지입니다.

카프카에서 복제된 파티션은 `리더 파티션`과 `팔로우 파티션`으로 구성된다.

프로듀서 or 컨슈머와 직접 통신하는 파티션을 `리터 파티션`이라고 부르며 그 외 복제 데이터를 가지고 있는 파티션을 `팔로워 파티션`이라고 부른다.

복제를 통하면 나머지 브로커에서도 파티션의 데이터가 복제되므로 복제 개수만큼 저장 용량이 증가한다는 단점이 있지만, 장점으로는 데이터를 안전하게 사용할 수 있다는 강력한 장점이 있다.

운영을 하다보면 뜻하지 않은 상황에서 `리더 파티션` 브로커에 장애가 발생 할 수 있다. 그러한 경우 복제로 사용되고 있는 팔로워 파티션을 가지고 있는 브로커중 하나가 `리더파티션`의 지위를 넘겨받는다. 이를 통해서
데이터가 유실되지 않고 컨슈머나 프로듀서와 데이터를 주고받도록 동작한다.

> 운영시 데이터가 일부 유실되어도 무관하고 데이터 처리속도가 중요하다면 1또는 2로 설정한다. <br>
> 금융권의 정보와 같이 유실이 일어나면 안되는 중요한 서비스에서는 최소 3개의 브로커로 구성하여 동시에 2개이상의 브로커가 장애가 발생하여도 나머지 하나로 안전하게 운영할 수 있도록 한다.

### 🚀 컨트롤러 (controller)

클러스터의 다수 브로커중 한 대가 컨트롤러의 역할을 한다. 컨트롤러는 다른 브로커들의 상태를 체크하고 브로커가 클러스터에서 빠지는 경우 해당 브로커에 존재하는 `리더 파티션`을 재 분배하는 역할을 한다. 카프카는
지속적으로 데이터를 처리해야 하기때문에 브로커의 상태가 비정상이라면 빠르게 클러스터에서 빼내는것이 중요하다. 컨트롤러역할을 하는 브로커에 장애가 생긴다면 다른 브로커가 컨트롤러 역할을 한다.

### 🚀 데이터 삭제

카프카는 다른 메시징 플랫폼과 다르게 컨슈머가 데이터를 가져가더라도 토픽의 데이터는 삭제되지 않는다. 또한 컨슈머나 프로듀서가 데이터 삭제를 요청할 수 없다.

오로지 브로커만이 데이터를 삭제할 수 있다. 데이터의 삭제는 파일단위로 이루어지며 이 단위를 `로그 세그먼트`라고 부른다. 로그 세그먼트에는 다수의 데이터가 들어가 있기 때문에 Database처럼 특정 데이터를
선별해서 삭제할 수 없다.

`로그 세그먼트`는 파일 단위로 이루어져 있음며, 데이터가 쌓이는 동안은 해당 파일 시스템으로 열려있어 카프카 브로커에
`log.segement.bytes` or `log.segment.ms` 옵션에 값이 설정되면 세그먼트 파일이 닫히게 된다.

`로그 세그먼트`가 닫히는 기본값은 1GB용량에 도달 하였을때이며, 옵션에 따라 더 작게 조절이 가능하나 너무 작게 설정하게되면 너무 파일을 여닫기 때문에 부하가 발생할 수 있어 주의해야한다.

닫힌 세그먼트 파일은 `log.retention.bytes` or `log.retention.ms`옵션에 설정값이 넘으면 삭제된다.

닫힌 세그먼트 파일을 체크하는 간격은 브로커의 옵션에 설정된 `log.retention.check.interval.ms`를 따른다

### 🚀 컨슈머 오프셋 저장

컨슈머 그룹은 파티션의 어드 레코드까지 데이터를 가져갔는지 확인하기 위해 오프셋을 커밋한다. 커밋한 오프셋은 __consumer_offsets 토픽에 저장된다, 저장된 오프셋을 토대로 컨슈머 그룹은 다음 레코드를
가져가서 처리한다.

------

### 🚀 코디네이터 ( coordinator )

클러스터의 다수 브로커 중 한대는 코디네이터의 역할을 수행한다. 코디네이터는 컨슈머 그룹의 상태를 체크하고 파티션을 컨슈머와 매칭되도록 분배하는 역할을 한다.

컨슈머가 컨슈머 그룹에서 빠지면 매칭되지 않은 파티션을 정상 동작한느 컨슈머로 할당하여 끊임없이 데이터가 처리되도록 도와준다.

이러한 파티션을 컨슈머로 재할당하는 과정을 `리벨런스`라고 부른다

## 🚀 토픽과 파티션

- 토픽 : 카프카에서 데이터를 구분하기 위해 사용 하는 단위 ( 토픽은 1개이상의 파티션을 소유한다 )
- 파티션에는 프로듀서가 보낸 데이터들이 들어가 저장된다. 이를 레코드라고 부른다.

레코드는 오프셋, 메시지 키 , 메시지 값으로 이루어져 있다.

파티션은 그룹으로 묶인 컨슈머들이 레코드를 병렬로 처리할 수 있도록 매칭해주며, 컨슈머의 처리량이 한정되 있는 경우 레코드를 병렬로 처리하는 좋은 방법은 컨슈머의 개수를 늘려 스케일 아웃 하는것이다.
( 토픽명이 같아도 파티션번호를 지정해 줄 수 있다.)

### 🚀 토픽명 제약 조건

- 토픽명은 공백 또는 빈 문자열일 수 없다.
- 토픽 이름은 마침표(.) 또는 마침표 둘(..)을 생성될 수 없다.
- 토픽 이름의 길이는 249자 미만으로 생성되야한다.
- 토픽명은 대,소문자, 0~9, 마침표(.), 언더바(_), 하이픈(-)조합으로만 생성가능.
- `__consumer_offsets` , `__transaction_state` 해당 두개의 토픽명으로는 생성 불가
- 마침표(.)와 언더바(_)가 동시에 들어가면 안된다, 내부적 로직때문에 사용은 가능하니 경고메시지 출력
- 이미 생성된 토픽에 마침표(.)와 언더바(_)가 있는데 해당 부분을 서로 변경시 이미 생성된 토픽이라고 한다.

### 🚀 토픽명 작명 템플릿 예시

- <환경>.<팀명>.<애플리케이션명>.<메시지타입>
- <프로젝트명>.<서비스명>.<환경>.<이벤트명>
- <환경>.<서비스명>.<JIRA-번호>.<메시지타입>
- <카프카클러스터명>.<환경>.<서비스명>.<메시지타입>

토픽명 규칙을 사전에 정의하지 않는다면 개개인이 쓰는 방식이 달라 기술 부채로 남아 나중에 운영시 힘들것이다. 또한 카프카는 토픽명의 변경을 지원하지 않기때문에 이름을 만들기 위해서는 삭제 후 다시 생성해야한다.

### 🚀 레코드

> 레코드는 타임스탬프, 메시지키, 메시지값, 오프셋, 헤더로 구성되어 있다. <br>
> 오프셋과 타임스탬프는 프로듀서가 생성한 레코드가 브로커로 전송되면 지정되어 저장되며, 한번 적재된 레코드는 수정할 수 없고 로그 리텐션 기간 또는 용량에 따라서만 삭제된다.

- 타임스탬프 : 프로듀서에서 해당 레코드가 생성된 시점의 유닉스 타임이 설정된다. 다만 프로듀서에서 임의로 타임스탬프 값을 설정할 수 있다. 추가로 브로커 설정에 따라 브로커에 적재된 시간으로도 설정 가능하다.<br>
<br>
- 메시지 키 : 메시지 값을 순서대로 처리하거나 메시지 값의 종류를 나타내기 위해 사용하며, 토픽에 레코드를 전송할때 해당 해시값을 토대로 파티션을 지정한다. 동일한 키라면 동일 파티션에 저장된다.
  만약 키를 지정하지 않는다면 null로 설정되며 기본 설정 파티셔너에 따라 저장된다.


- 메시지 값 : 실질적으로 처리하고자 하는 데이터가 들어간다. 직렬화한 형태와 동일한 형태로 역직렬화 해야하며, 기본 직렬화 외에도 옵션을 통하여 데이터 직렬화,역직렬화 가능
- 오프셋 : 0이상의 숫자로 이루어져 있으며, 컨슈머가 데이터를 가져갈 때 사용된다. 해당 값을 통해 파티션의 데이터를 어디까지 가져갔는지 지정할 수 있다.

### 🚀 프로듀서의 주요 옵션

#### 필수 옵션

> ● bootstrap.servers: 프로듀서가 데이터를 전송할 대상 카프카 브로커의 호스트를 입력한다 (다중입력 가능)
> ● key.serializer: 레코드의 메시지 키를 직렬화하는 클래스를 지정한다.
> ● value.serializer: 레코드의 메시지 값을 직렬화하는 클래스를 지정한다.

<br>

#### 선택 옵션
> ● acks: 프로듀서가 전송한데이터가 브로커에 정상적으로 저장되었는지 성공 여부를 확인하는데 사용 0,1(default),-1로 설정 가능<br>
> <b>acks option</b> 1 : 리더 파티션에 데이터가 저장되면 정상 전송으로 판단. <br>
> <b>acks option</b> 0 : 프로듀서가 전송한 즉시 성공으로 판단  <br>
> <b>acks option</b> -1 : min.insync.replicas개수에 해당하는 리더파티션과 팔로우 파티션에 데이터가 저장되면 성공으로 판단.<br>
> java code에서 send(record).get()으로 확인 가능 하지만 get()은 동적으로 동작하기 때문에 Callback클래스를 재구현하여 비동기로 결과를 확인 할 수 있도록 클래스를 제공한다.
> 
> ● buffer.memory: 브로커로 전송할 데이터를 배치로 모으기 위한 버퍼 메모리양 설정 기본값은 32MB.<br><br>
> ● retries: 프로듀서가 브로커로부터 에러를 받고 난 뒤 재전송을 시도하는 횟수 기본값은 2147483647(21억..) <br><br>
> ● batch.size: 배치로 전송할 레코드  최대 용량 지정. 기본값은 16384이며 너무 적게 설정하면 자주 전송하기때문에 네트워크 부담이 너무 크고, 크게 설정하면 메모리를 더 많이 사용하게 된다는 점이있다. <br><br>
> ● linger.ms : 배치를 전송하기 전까지 기다리는 최소 시간, default 0 <br><br>
> ● partitioner.class: 레코드를 파티션에 전송할 때 적용하는 파티셔너 클래스를 지정 기본값은 DefaultPartitioner <br><br>
> ● enable.idempotence: 멱등성 프로듀서 동작 유무 설정 <br><br> 
> ● transactional.id: 프로듀서가 레코드를 전송할 때 레코드를 트랜잭션 단위로 묶을지 여부 설정. default null

### 🚀 컨슈머의 중요 개념
> 1개이상의 컨슈머로 이루어진 컨슈머 그룹을 운영하는 방식과 토픽의 특정 파티션만 구독하는 컨슈머를 운영하는것이다.

컨슈머 그룹의 컨슈머 개수는 파티션의 개수보다 작거나 같아야 한다. 하나의 파티션은 하나의 컨슈머만 바라볼 수 있는데
파티션의 갯수보다 컨슈머의 갯수가 많다면 유휴 상태로 남는 컨슈머가 생겨 실질적인 데이터 처리를 하지 못하게 되므로 불필요한 스레드로 남게 된다.

> 이미지 3.4.2-5에서는하나의 파티션이 다른 컨슈머 그룹에 할당되는것을 보니 같은 컨슈머그룹에 하나씩 매칭 가능한듯.  


컨슈머 그룹은 다른 컨슈머 그룹과 격리되는 특징을 가지고있다. 이말은 즉 카프카 프로듀서가 보낸 데이터를 각기
다른 역할을 하는 컨슈머 그룹까지 영향을 받지 않게 처리할 수 있다는 장점이 있다.

### 🚀 컨슈머의 주요 옵션

#### 필수옵션
 - 필수옵션은 프로듀서의 필수 옵션과 동일


#### 선택 옵션
> <b>group.id</b> : 컨슈머 그룹 아이디를 지정한다 subscribe()메서드로 토픽을 구독하면 해당 옵션을 필수로 넣어야 한다 default null <br>
> 
> <b>auto.offset.reset</b> : 컨슈머 그룹이 특정 파티션을 읽을때 지정한 컨슈머 오프셋이 없는 경우 어느 오프셋부터 읽을지 선택하는 옵션. 이미 있다면 해당 옵션은 무시 (latest, earliest, none)중 설정 가능 default latest
> <br> - latest : 가장 높은 오프셋 부터 읽기 시작(최신 레코드)
> <br> - earliest : 가장 낮은 오프셋부터 읽기 시작한다 (오래된 레코드)
> <br> - none : 컨슈머 그룹이 커밋한 기록이 있는지 찾아본다, 기록이 없다면 오류 반환, 있다면 기존 커밋 기록 이후 오프셋부터 읽기 시작한다
> 
> enable.auto.commit : 자동 커밋할 지 수동으로 커밋할지 선택 가능. default true
> 
> auto.commit.interval.ms : 자동 커밋일 경우 오프셋 커밋 간격일 지정 default 5000ms (5초)
> 
> max.poll.records: poll() 메서드를 통해 반환되는 레코드 개수를 지정 default 500
> 
> session.timeout.ms : 컨슈머가 브로커와 연결이 끊기는 최대 시간, 해당 옵션의 시간동안 하트비트를 전송하지 않으면 브로커는 컨슈머에 이슈가 밸생했다고 가정하고 리밸런싱을 시작한다 default 10000ms(10초) 기본적으로 `heartbeat.interval.ms` 시간의 3배이상으로 설정한다.
> 
> heartbeat.interval.ms: 하트비트를 전송하는 시간 간격 default 3000ms (3초) 오탈자 조심
> 
> max.poll.interval.ms : pool()메서드를 호출하는 간격의 최대 시간을 지정, 만약 오래걸릴 경우 비정상으로 판단하고 리밸런싱 시작, default 300000ms(5분)
> 
> isolation.level : 트랜잭션 프로듀서가 레코드를 트랜잭션 단위로 보낼 경우 사용 default read_uncommitted
